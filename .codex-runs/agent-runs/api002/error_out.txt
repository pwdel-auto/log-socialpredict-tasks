**Recommendation**

Adopt a small boundary-owned `reason` vocabulary that is stable, public, and transport-scoped, then map existing typed/internal errors to that vocabulary only at the HTTP serialization boundary. Keep domain and repository layers free to use richer internal error detail; handlers should only choose status + public `reason` + sanitized message/body shape through shared helpers.

Recommended initial vocabulary, kept intentionally small:

- `invalid_request`
- `unauthenticated`
- `forbidden`
- `not_found`
- `conflict`
- `rate_limited`
- `unprocessable`
- `dependency_unavailable`
- `internal`

This should likely live beside existing HTTP error serialization code, not in domain packages. The smallest defensible change is probably in `backend/errors/httperror.go` plus focused handler-facing helper coverage and one route-visible doc surface in `backend/docs/openapi.yaml`.

**Error Taxonomy Plan**

- Define public constants for the stable `reason` values in the HTTP/boundary error package.
- Add a boundary helper that converts existing internal/typed errors into `(status, reason, safe body fields)`.
- Preserve existing response envelope shape if one already exists; add `reason` rather than redesigning error responses.
- Treat `reason` as API contract, not as a dump of Go error types or DB errors.
- Default unknown/unclassified failures to `internal`.
- Avoid adding operation-specific reasons unless already required by route behavior; prefer stable common classes.

**Boundary Mapping Table**

| Internal failure class | HTTP status | Public `reason` | Notes |
|---|---:|---|---|
| Validation/input parse failure | 400 | `invalid_request` | Bad query/body/path/header syntax or missing required input |
| Authentication missing/invalid | 401 | `unauthenticated` | Do not reveal token/session specifics |
| Authenticated but not allowed | 403 | `forbidden` | Covers ownership/admin/policy denial |
| Missing resource/entity | 404 | `not_found` | Avoid existence leaks where policy requires 403 |
| Duplicate/state collision | 409 | `conflict` | Unique constraint or incompatible state transition |
| Semantic business-rule rejection | 422 | `unprocessable` | Valid syntax, rejected by domain rule |
| Rate/security throttling | 429 | `rate_limited` | Keep retry details minimal unless already standard |
| Upstream/service temporarily unavailable | 503 | `dependency_unavailable` | For setup/dependency/load failures visible at boundary |
| Unexpected/unclassified failure | 500 | `internal` | Never expose internals |

**Important Risks / Edge Cases**

- `403` vs `404` is the highest leakage risk. Some routes should not confirm resource existence.
- `400` vs `422` can drift if handlers currently mix parse errors and domain validation failures.
- Unique-constraint or DB-state errors must not leak raw DB strings while still mapping consistently to `conflict`.
- Rate-limit and auth middleware may already emit responses outside handler-local code paths; those paths must share the same `reason` vocabulary or be explicitly exempted.
- If some handlers already return bespoke error JSON, introducing `reason` inconsistently will create contract drift.
- OpenAPI should document the common vocabulary without falsely claiming every route returns every reason.

**Leakage And Sanitization Risk Register**

- Raw database errors: must never surface in `message`; map to `conflict` or `internal`.
- Auth/session/token failures: never distinguish expired vs unknown vs malformed unless already public contract.
- Permission denials: prefer `forbidden`; use `not_found` where existence itself is sensitive.
- Internal dependency/setup errors: expose only `dependency_unavailable` or `internal`, not configuration names, URLs, or secrets.
- Domain/business errors: expose stable category only; keep market/accounting internals out of free-text messages.
- Panic/unhandled errors: force `internal`, log privately, sanitize response body.
