Recommendation

Adopt this as a shared boundary contract, not a domain contract: define the public `reason` vocabulary once in `backend/errors/` (or an adjacent boundary-owned helper under `backend/handlers/...` only if the repo already keeps transport DTO helpers there), keep `reason` optional for backward compatibility, and document it through the shared OpenAPI error schema used by route responses. The smallest defensible change is:

- add public constants plus a narrow mapping helper at the HTTP boundary
- map only common transport-visible failure classes, not domain-specific business cases
- update focused serialization tests in `backend/errors/httperror_test.go` and one or two route-facing tests where the reason is emitted
- update `backend/docs/openapi.yaml` so the shared error response schema includes `reason` and describes the stable vocabulary and unknown-value tolerance

Important risks and edge cases

- Main compatibility risk: making `reason` required would be a wire break for existing clients.
- Secondary compatibility risk: using a closed OpenAPI `enum` for `reason` makes future additions breaking for strict/generated clients. Safer contract: `type: string`, examples, documented known values, and explicit “clients must tolerate unknown values”.
- Boundary leak risk: handlers should not infer detailed business semantics just to produce a prettier `reason`. Only map already-classified outcomes from existing errors.
- Consistency risk: if some handlers still write ad hoc JSON errors, the vocabulary will fragment immediately.
- Auth edge case: `401` and `403` must not collapse into one reason; keep unauthorized vs forbidden distinct.
- Validation edge case: request-shape/parse failures, missing required inputs, and semantic invalid state should not all share one bucket unless the existing error model already does that.

Contract compatibility assessment

Conditionally compatible and low-risk if `reason` is additive and optional. High-risk if the change also normalizes previously inconsistent response shapes or introduces a required field. From an OpenAPI perspective, prefer:

- additive property on the existing shared error schema
- `string` rather than closed `enum`
- examples for the initial vocabulary
- explicit note that unknown future values are allowed
